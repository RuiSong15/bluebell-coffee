# 🫐 Bluebell Coffee
### WEB700 – Web Programming | Project Part 3 | Team Blue

**Business idea:** Bluebell Coffee is a specialty coffee shop. The application lets staff browse, search, and manage the drink menu and its order records.
**Target users:** Shop staff (menu management) and customers (browsing/searching the menu).

---

## How Part 3 Extends Part 2

| Part | Data source | Capability |
|------|-------------|-----------|
| Part 2 | Local `data/drinks.json` file | Read-only routes, search, form validation |
| Part 3 | **Neon PostgreSQL via Sequelize** | Full CRUD (add / edit / remove menu records), health check, deployed to Vercel |

The local JSON file is **no longer used at runtime** — it is kept only as the source for the one-time database import (`scripts/seed.js`).

**Dataset:** 521 order records covering 24 menu items in 5 categories, regenerated for Part 3 with the identical field structure as Part 2 (`scripts/generate-data.js`, deterministic/reproducible). Every menu item has its own SVG illustration in `public/images/` plus a hero banner, generated by `scripts/generate-images.js` (no copyright issues, works offline and on Vercel).

## Database Design (Neon PostgreSQL)

One main table: **`drinks`** (521 imported records).

| Column | Type | Constraints |
|--------|------|-------------|
| id | SERIAL | Primary key |
| order_id | VARCHAR(20) | |
| item_id | VARCHAR(20) | |
| name | VARCHAR(100) | NOT NULL |
| category | VARCHAR(50) | NOT NULL |
| size | VARCHAR(20) | NOT NULL, default `'N/A'` |
| price | DECIMAL(6,2) | NOT NULL, ≥ 0 |
| quantity | INTEGER | NOT NULL, ≥ 1, default 1 |
| order_type | VARCHAR(20) | NOT NULL, one of Takeout / Dine-in / Delivery |
| sku | VARCHAR(20) | |
| customer | VARCHAR(50) | |
| popularity | INTEGER | NOT NULL, ≥ 0, default 0 |
| ingredients | JSONB | NOT NULL, default `[]` |
| created_at | TIMESTAMP | NOT NULL, default now |

### JSON → Column Mapping

| JSON field | Database column | Decision |
|------------|-----------------|----------|
| `id` ("order-001") | *(dropped)* | Replaced by auto-increment integer `id` |
| `order_id`, `item_id`, `name`, `category`, `size`, `price`, `quantity`, `order_type`, `created_at` | same names | Direct mapping with proper types |
| `details.sku` / `details.customer` / `details.popularity` | `sku` / `customer` / `popularity` | **Flattened** nested object into normal columns |
| `ingredients` (array) | `ingredients` | Stored as **JSONB** (genuinely list-shaped data) |

### Sequelize Model

`models/Drink.js` maps to the `drinks` table (`tableName: 'drinks'`, `timestamps: false`) with model-level validation (required fields, non-negative price, quantity ≥ 1, order-type whitelist). One reusable connection is created in `config/database.js` and shared by all routes. No destructive sync runs at startup.

## Routes

| Method | URL | Output | Purpose |
|--------|-----|--------|---------|
| GET | `/` | HTML | Home page: hero banner + illustrated menu cards (fetches `/api/drinks`, groups orders into menu items, client-side search/sort) |
| GET | `/drinks` | EJS HTML | Menu table from PostgreSQL, category filter |
| GET | `/search` | EJS HTML | Search form |
| POST | `/search` | EJS HTML | Search results (validated) |
| GET | `/drinks/add` | HTML form | Insert form |
| POST | `/drinks/add` | Redirect | Validate + INSERT new record |
| GET | `/drinks/:id/edit` | HTML form | Prepopulated edit form |
| POST | `/drinks/:id/edit` | Redirect | Validate + UPDATE record |
| POST | `/drinks/:id/delete` | Redirect | DELETE record (confirm step in UI) |
| GET | `/api/drinks` | JSON | All records via Sequelize |
| GET | `/api/drinks/:id` | JSON | One record by ID |
| GET | `/api/search?keyword=` | JSON | Search by keyword |
| GET | `/health` | JSON | DB connectivity + record count |
| * | anything else | 404 HTML/JSON | Unknown-route handler |

Error handling covers: missing `DATABASE_URL`, DB connection failure, malformed IDs, records not found, missing/invalid field values, Sequelize validation errors, and unexpected server errors (no stack traces or credentials are exposed).

## Local Setup

```bash
npm install
copy .env.example .env      # then paste your Neon connection string into .env
npm run seed                # one-time import of the 521 JSON records
npm start                   # http://localhost:3000
```

**Environment variables** (see `.env.example`): `DATABASE_URL` (required), `PORT` (optional).

### Data Import
`scripts/seed.js` creates the table if needed, refuses to run if data already exists (so it can never wipe production data), maps every JSON record to the column layout above, and bulk-inserts with validation.

### Testing
- `GET /health` should return `{"status":"ok","database":"connected","records":521}`
- Add / edit / delete a test record through the UI, then verify the change in pgAdmin.

## Deployment (Vercel)

1. Push to the shared GitHub repository.
2. Import the repo in Vercel (`vercel.json` is already configured).
3. Add the environment variable `DATABASE_URL` using the **pooled** Neon connection string.
4. Deploy, then test `/health` and all routes on the deployed URL.

## Known Limitations

- Single-table design; ingredients are not normalized into a separate table.
- No authentication — menu management is open to anyone with the URL.
- Success messages are passed via query string rather than sessions.
